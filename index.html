Key Facts About the Stack

1
The stack grows DOWNWARD in memory (PUSH decreases ESP by 4, POP increases it by 4). In this view, we show the top of stack at the visual top — addresses get higher as you go down.
2
ESP always points to the top of the stack — the lowest-address used slot, shown at the visual top. Reading from [ESP] gives you the most recently pushed value.
3
EBP is your frame anchor. After the prologue, [ebp-4] is your first local variable and [ebp+8] is your first function argument.
4
CALL pushes the return address automatically. RET pops it back. This is how functions return to where they were called from.
5
LIFO (Last In, First Out): if you push 10, 20, 30 and pop three times, you get 30, 20, 10 in that order.

Two's Complement

How computers represent negative numbers. This is the most important number concept in assembly.

The Problem

Registers hold raw bits. There's no "negative sign" — just 0s and 1s. So how do we represent -5?

Two's complement is the convention: the high bit acts as the sign bit. If it's 1, the number is negative.

How It Works (8-bit example)

+5:0 0 0 0 0 1 0 1= 5
NOT:1 1 1 1 1 0 1 0= 250 (flip all bits)
+1:1 1 1 1 1 0 1 1= 251 unsigned, -5 signed

To negate: flip all bits, then add 1. That's exactly what the NEG instruction does.

To go back: flip and add 1 again (the process is its own inverse).

The Range

SizeUnsigned RangeSigned Range
8-bit0 to 255-128 to 127
16-bit0 to 65,535-32,768 to 32,767
32-bit0 to 4,294,967,295-2,147,483,648 to 2,147,483,647

The same bits can be interpreted as signed or unsigned. The CPU doesn't care — it's your instructions (DIV vs IDIV, JA vs JG) that determine the interpretation.

Key Patterns to Memorize

Pattern (8-bit)UnsignedSignedNote
0000000000Zero
01111111127+127Max positive
10000000128-128Min negative (sign bit flips!)
11111111255-1All bits set = -1
11111110254-2
11111011251-5

Why Two's Complement?

Because addition just works. The CPU uses the same ADD circuit for signed and unsigned numbers:

3:0 0 0 0 0 0 1 1
-5:1 1 1 1 1 0 1 1
Sum:1 1 1 1 1 1 1 0= 254 unsigned = -2 signed

3 + (-5) = -2. It works! The carry out of bit 8 is simply discarded.

  • What is 0xFFFFFFFB as a signed decimal? (Should be -5)
  • What is 128 in signed 8-bit? (Should be -128 — it overflows!)

CPU Flags

Flags are single-bit values the CPU sets automatically after most operations. Conditional jumps read them.

The Four Key Flags

FlagNameSet to 1 when...Example
ZFZero FlagResult is exactly 0sub eax, eax → ZF=1
SFSign FlagResult's highest bit is 1 (negative if signed)sub eax, ebx when eax < ebx
CFCarry FlagUnsigned overflow/underflow (borrow)sub 3, 5 → CF=1 (unsigned borrow)
OFOverflow FlagSigned overflow (result doesn't fit)add 127, 1 in 8-bit → OF=1

Which Instructions Set Flags?

DO set flags

  • ADD, SUB, INC, DEC
  • AND, OR, XOR
  • NEG, NOT
  • CMP (= SUB that discards result)
  • TEST (= AND that discards result)
  • SHL, SHR, SAR
  • MUL, IMUL, DIV, IDIV

Do NOT set flags

  • MOV
  • LEA
  • PUSH, POP
  • CALL, RET
  • XCHG
  • CDQ
  • NOP
This is critical for branching! If you see MOV between a CMP and a JE, the MOV does NOT change the flags — the jump still uses the flags from the CMP.

CF vs OF: Unsigned vs Signed Overflow

CF (Carry Flag) = unsigned arithmetic went out of range.

OF (Overflow Flag) = signed arithmetic went out of range.

The CPU sets both flags on every arithmetic operation. Your choice of jump instruction determines which one matters:

Unsigned jumps (use CF)Signed jumps (use OF/SF)
JA, JAE, JB, JBEJG, JGE, JL, JLE
Above / BelowGreater / Less

Memory aid: Above/Below = unsigned. Greater/Less = signed.

How CMP Sets Flags

CMP A, B computes A - B internally and sets flags from the result:

ConditionZFCFJump that fires
A == B10JE / JZ
A != B0?JNE / JNZ
A < B (unsigned)01JB
A ≤ B (unsigned)??JBE (CF=1 or ZF=1)
A > B (unsigned)00JA (CF=0 and ZF=0)

Memory Addressing & Data Sizes

Assembly works with raw memory. You need to know how to read addresses and how much data each operation moves.

Data Sizes

NameSizeRegistersDirective
BYTE1 byte (8 bits)AL, BL, CL, DLbyte ptr
WORD2 bytes (16 bits)AX, BX, CX, DXword ptr
DWORD4 bytes (32 bits)EAX, EBX, ECX, EDXdword ptr
QWORD8 bytes (64 bits)EDX:EAX (pair)qword ptr

When you see dword ptr [ebp-8], it means: access 4 bytes of memory starting at address ebp-8.

Square Brackets = Memory Access

Square brackets [ ] mean "go to this address in memory and read/write there":

  • mov eax, ebx — copy the value in EBX to EAX (register to register)
  • mov eax, [ebx] — go to the address stored in EBX, read 4 bytes from memory, put them in EAX
  • mov [ebx], eax — write EAX's value to the memory address in EBX
No brackets = register value. Brackets = memory at that address. This is the most fundamental distinction in assembly operands.

Common Addressing Modes

SyntaxMeaningCommon Use
[ebp-4]Memory at EBP minus 4Local variable
[ebp+8]Memory at EBP plus 8Function argument
[eax]Memory at address in EAXPointer dereference
[eax+ecx]Base + indexArray access
[eax+ecx*4]Base + index * scaleArray of DWORDs
[eax+ecx*4+8]Base + index * scale + offsetStruct array member

When You See "dword ptr"

The ptr prefix tells the assembler the size of the memory access when it can't be inferred from the registers:

mov dword ptr [ebp-8], 45   ; Write 4 bytes (32-bit value 45)
mov byte ptr [ebp-4], 0xFF  ; Write 1 byte
mov word ptr [ebp-2], 1000  ; Write 2 bytes

If a register operand is present, the size is usually inferred: mov [ebp-8], eax is implicitly dword because EAX is 32-bit.

IDA Stack Variable Notation

In IDA Pro, you'll see named variables instead of raw offsets:

[ebp+var_4]   = [ebp-4]    ; IDA names local var at offset -4
[ebp+var_8]   = [ebp-8]    ; local var at offset -8
[ebp+arg_0]   = [ebp+8]    ; first function argument
[ebp+arg_4]   = [ebp+12]   ; second function argument

The "var_" prefix means local variable (negative offset). The "arg_" prefix means function argument (positive offset past saved EBP and return address).

Arithmetic Instructions

These are the building blocks. Every assembly program uses these constantly.

MOV — The Foundation

MOV dest, src copies a value into a register. The source is unchanged.

Think of it like assignment in a programming language: dest = src

Try it:

ADD & SUB

ADD dest, src → dest = dest + src

SUB dest, src → dest = dest - src

The result is stored in the destination (first operand). The source is unchanged.

When SUB produces a negative result in an unsigned register, it wraps around (e.g., 5 - 10 in 32-bit = 4294967291, which is -5 in two's complement).
Try it:

INC, DEC, NEG

INC dest → dest = dest + 1

DEC dest → dest = dest - 1

NEG dest → dest = -dest (two's complement negation: flip bits, add 1)

Bitwise Logic Instructions

These operate on individual bits. To understand them, you need to think in binary.

AND — Keep only shared bits

AND dest, src: A result bit is 1 only if BOTH input bits are 1.

Common use: masking out bits. AND al, 0x0F keeps only the lower 4 bits.

A:1 0 1 1= 11
B:1 1 0 1= 13
AND:1 0 0 1= 9
Try it (8-bit):

OR — Combine bits

OR dest, src: A result bit is 1 if EITHER input bit is 1.

Common use: setting bits. OR al, 0x80 sets the high bit.

A:1 0 1 0= 10
B:0 1 1 0= 6
OR:1 1 1 0= 14

XOR — Toggle bits

XOR dest, src: A result bit is 1 if the inputs are DIFFERENT.

Key pattern: XOR reg, reg = 0 (fastest way to zero a register).

A:1 1 0 1= 13
B:1 0 1 1= 11
XOR:0 1 1 0= 6
Try it (8-bit):

NOT — Flip all bits

NOT dest: Flips every bit (0→1, 1→0). Also called one's complement.

For 8-bit: NOT x = 255 - x. For 32-bit: NOT x = 0xFFFFFFFF - x.

A:0 0 0 0 0 0 1 1= 3
NOT:1 1 1 1 1 1 0 0= 252

TEST — AND without storing

TEST a, b computes a AND b but throws away the result. Only the flags are set.

Most common: TEST reg, reg checks if a register is zero (sets ZF=1 if zero).

Multiply & Divide

These are trickier because they use implicit registers (EDX:EAX) and have signed/unsigned variants.

MUL — Unsigned Multiply

MUL src multiplies EAX by src. The 64-bit result goes into EDX:EAX.

  • EAX = low 32 bits of the result
  • EDX = high 32 bits (often 0 if the result fits in 32 bits)
MUL always clobbers EDX! Even if EDX had a value you cared about, it gets overwritten.
Try it: 100 * 5

DIV — Unsigned Divide

DIV src divides EDX:EAX by src.

  • EAX = quotient (result of division)
  • EDX = remainder (modulo)
You MUST zero EDX before DIV (unless you intentionally want a 64-bit dividend). Use xor edx, edx or mov edx, 0.
Try it: 17 / 5 = 3 remainder 2

IMUL — Signed Multiply

IMUL has three forms:

  • IMUL src — EDX:EAX = EAX * src (one operand, like MUL but signed)
  • IMUL dest, src — dest = dest * src (two operands)
  • IMUL dest, src, imm — dest = src * imm (three operands)

The 2- and 3-operand forms do NOT affect EDX.

IDIV & CDQ — Signed Divide

IDIV src is signed division of EDX:EAX by src.

Before IDIV, use CDQ to sign-extend EAX into EDX:EAX:

  • If EAX ≥ 0: EDX becomes 0x00000000
  • If EAX < 0: EDX becomes 0xFFFFFFFF
Try it: 7168 / (-65)

16-bit MUL and DIV

When using 16-bit operands (e.g., div cx), the behavior changes:

  • MUL cx: DX:AX = AX * CX
  • DIV cx: AX = DX:AX / CX, DX = remainder

Notice it uses DX:AX (16-bit halves) instead of EDX:EAX.

Shift Operations

Shifts move bits left or right. They're closely related to multiplication and division by powers of 2.

SHL — Shift Left

SHL dest, count: Shifts all bits left. Zeros fill from the right.

Equivalent to: dest = dest * 2^count

Before:0 0 0 0 1 0 1 0= 10
SHL 2:0 0 1 0 1 0 0 0= 40

10 * 2^2 = 10 * 4 = 40

SHR — Shift Right (Logical)

SHR dest, count: Shifts all bits right. Zeros fill from the left.

Equivalent to: dest = dest / 2^count (unsigned, rounds down)

Before:0 0 1 0 1 0 0 0= 40
SHR 3:0 0 0 0 0 1 0 1= 5

40 / 2^3 = 40 / 8 = 5

Try it: SHR then SHL

SAR — Shift Right (Arithmetic)

SAR dest, count: Like SHR but preserves the sign bit.

If the number is negative, ones fill from the left (keeps it negative). This is signed division by 2^count.

SHR vs SAR matters for negative numbers! SHR treats everything as unsigned. SAR preserves the sign.

Branching (Jumps)

Branching lets code make decisions. It works in two steps: set flags, then jump based on flags.

Step 1: Set Flags with CMP or TEST

CMP a, b computes a - b and sets flags (but doesn't store the result).

TEST a, b computes a AND b and sets flags (but doesn't store the result).

FlagNameSet when...
ZFZero FlagResult is zero (a == b for CMP)
CFCarry FlagUnsigned borrow/carry occurred (a < b unsigned for CMP)
SFSign FlagResult is negative (high bit is 1)
OFOverflow FlagSigned overflow occurred

Step 2: Conditional Jump

InstructionConditionAfter CMP a, b: jumps if...
JE / JZZF = 1a == b
JNE / JNZZF = 0a != b
JB / JNAECF = 1a < b (unsigned)
JBE / JNACF=1 or ZF=1a ≤ b (unsigned)
JA / JNBECF=0 and ZF=0a > b (unsigned)
JAE / JNBCF = 0a ≥ b (unsigned)
JL / JNGESF != OFa < b (signed)
JLE / JNGZF=1 or SF!=OFa ≤ b (signed)
JG / JNLEZF=0 and SF==OFa > b (signed)
JMPAlways(unconditional)

Common Pattern: TEST for zero

TEST reg, reg followed by JNZ checks if a register is nonzero.

TEST reg, reg followed by JZ checks if a register is zero.

This is because reg AND reg = reg, and ZF is set if the result is 0.

How to trace branches

  1. Track register values up to the CMP/TEST
  2. Determine the flag values from the comparison
  3. Check the jump condition against those flags
  4. If taken: skip to the target label. If not: continue to next line.
  5. Keep tracking registers along whichever path was taken

For your worksheet, write the register values at each step AND mark each jump as "TAKEN" or "NOT TAKEN".

Stack & Functions

The stack is how x86 manages function calls, local variables, and arguments.

The Stack Grows Downward

ESP (Stack Pointer) points to the top of the stack. The stack grows toward lower addresses.

  • PUSH val: ESP -= 4, then store val at [ESP]
  • POP dest: Load [ESP] into dest, then ESP += 4
Stack (growing downward)
Higher addr ...
[EBP+12] arg_2 3rd argument
[EBP+8] arg_1 2nd argument (first pushed)
[EBP+4] return addr CALL pushed this
[EBP+0] old EBP saved base pointer ← EBP points here
[EBP-4] local_1 1st local variable
[EBP-8] local_2 2nd local variable
[ESP] ... ← ESP points here (top of stack)

Function Prologue & Epilogue

Almost every function starts and ends the same way:

; Prologue - set up stack frame
push ebp          ; Save caller's base pointer
mov ebp, esp      ; Set up our base pointer
sub esp, N        ; Allocate N bytes for local variables

; ... function body ...

; Epilogue - clean up
mov esp, ebp      ; Deallocate locals (or use LEAVE)
pop ebp           ; Restore caller's base pointer
ret               ; Return to caller

Reading Stack Offsets

Once you know the prologue pattern, reading stack variables is straightforward:

  • [ebp-4], [ebp-8], etc. = local variables
  • [ebp+0] = saved old EBP
  • [ebp+4] = return address
  • [ebp+8] = first argument
  • [ebp+12] = second argument, and so on

CALL and RET

CALL func: Pushes the return address (next instruction) onto the stack, then jumps to func.

RET: Pops the return address from the stack and jumps to it.

After a CALL returns, the caller cleans up pushed arguments with add esp, N (cdecl convention).

Tips for Stack Tracing

  1. Draw the stack on paper! Label each slot with its address (ebp-4, ebp-8, etc.) and current value.
  2. When a CALL happens, note what was pushed as arguments (right-to-left).
  3. In the called function, map [ebp+8] to the first argument, [ebp+12] to the second, etc.
  4. Track the return value in EAX.
  5. After returning, the caller stores EAX somewhere and cleans up ESP.

Common Assembly Idioms

Experienced assembly readers recognize these patterns instantly. Learn them and you'll read code much faster.

Zeroing a Register

xor eax, eax    ; EAX = 0 (faster than mov eax, 0)

XOR anything with itself gives 0. Compilers use this because it's 2 bytes vs 5 for mov eax, 0. When you see xor reg, reg, just read it as "reg = 0".

Test for Zero

test eax, eax   ; Sets ZF=1 if EAX is zero
jnz not_zero    ; Jump if EAX != 0
jz  is_zero     ; Jump if EAX == 0

TEST reg, reg = "is this register zero?" It ANDs the register with itself (result = the register) and sets ZF. More efficient than cmp eax, 0.

LEA for Math

lea eax, [ebx+ecx*4+8]  ; EAX = EBX + ECX*4 + 8

LEA computes an address but doesn't access memory. Compilers abuse this for math: it can do addition, scaling (x2, x4, x8), and addition of a constant — all in one instruction, without affecting flags.

lea eax, [eax+eax*4]    ; EAX = EAX * 5
lea eax, [ecx+edx]      ; EAX = ECX + EDX (without modifying ECX or EDX)

CDQ Before IDIV

cdq            ; Sign-extend EAX into EDX:EAX
idiv ecx       ; Signed divide EDX:EAX by ECX

Always see CDQ right before IDIV. It's required because IDIV divides the 64-bit value EDX:EAX, and CDQ sets EDX correctly based on EAX's sign. Similarly, xor edx, edx before unsigned DIV.

Function Prologue / Epilogue

; Prologue (start of every function)
push ebp         ; Save caller's frame
mov ebp, esp     ; Set up our frame
sub esp, N       ; Allocate local variables

; Epilogue (end of every function)
mov esp, ebp     ; Deallocate locals (or: leave)
pop ebp          ; Restore caller's frame
ret              ; Return

When you see push ebp / mov ebp, esp, you know a function just started. When you see pop ebp / ret, it's ending.

Multiply by Shift

shl eax, 1   ; EAX *= 2
shl eax, 2   ; EAX *= 4
shl eax, 3   ; EAX *= 8

shr eax, 1   ; EAX /= 2 (unsigned)
sar eax, 1   ; EAX /= 2 (signed, preserves sign)

Compilers often replace multiply/divide by powers of 2 with shifts. Faster and equivalent.

Move-and-Test Pattern

mov eax, [ebp-4]   ; Load a value
test eax, eax      ; Is it zero?
jz skip            ; If zero, skip ahead

Note: MOV does NOT set flags, so the TEST is needed. You'll see this pattern after loading function return values or struct fields.

Calling Convention Cleanup (cdecl)

push arg3       ; Push arguments right-to-left
push arg2
push arg1
call function
add esp, 12     ; Caller cleans up 3 args * 4 bytes = 12

The add esp, N after a CALL removes the pushed arguments. N / 4 = number of arguments pushed.

Functional Analysis

One of the most important RE skills: read a function's assembly and describe what it does in plain terms like f(a, b) = a + b.

The Approach

When analyzing a function, work from the simplest functions first, then use those to understand more complex ones that call them.

  1. Identify the prologue/epilogue to find where the function starts and ends
  2. Count the arguments by looking at [ebp+8], [ebp+12], etc. (or IDA's arg_0, arg_4)
  3. Track the return value — it's always in EAX when ret executes
  4. Trace the operations on the arguments to see what math/logic is performed
  5. Write it as a formula using the arguments as variables

Example: Simple Multiply Function

Func5:
  push ebp
  mov ebp, esp
  mov eax, [ebp+arg_0]    ; Load first argument (a)
  imul eax, [ebp+arg_4]   ; Multiply by second argument (b)
  pop ebp
  ret                      ; Return value in EAX = a * b
Analysis:
  • 2 arguments: arg_0 (a) and arg_4 (b)
  • EAX = a, then EAX = a * b
  • Returns EAX = a * b
Func5(a, b) = a * b

Example: Simple Divide Function

Func4:
  push ebp
  mov ebp, esp
  mov eax, [ebp+arg_0]    ; Load first argument (a)
  mov edx, 0              ; Zero EDX before DIV
  div [ebp+arg_4]         ; Unsigned divide by second argument (b)
  pop ebp
  ret                      ; Return value in EAX = a / b
Analysis:
  • 2 arguments: a and b
  • EAX = a / b (unsigned integer division, quotient only)
Func4(a, b) = a / b

Example: Function That Calls Other Functions

Once you know what the simple functions do, you can substitute:

Func2:
  push ebp
  mov ebp, esp
  sub esp, 16
  push [ebp+arg_8]         ; Push c (third arg)
  push [ebp+arg_4]         ; Push b (second arg)
  call Func4               ; Func4(b, c) = b / c
  add esp, 8               ; Clean up 2 args
  mov [ebp+var_4], eax     ; var_4 = b / c
  mov edx, [ebp+arg_0]     ; EDX = a (first arg)
  sub edx, eax             ; EDX = a - (b / c)
  mov eax, edx             ; Return value = a - b/c
  leave
  ret
Analysis:
  • 3 arguments: a, b, c
  • Calls Func4(b, c) which we know returns b / c
  • Subtracts that result from a
Func2(a, b, c) = a - (b / c)

Reading the Calling Convention

Arguments are pushed right-to-left in cdecl. So the push order tells you which argument is which:

push [ebp+var_8]    ; This becomes arg_2 (third parameter)
push [ebp+var_C]    ; This becomes arg_1 (second parameter)
push [ebp+var_14]   ; This becomes arg_0 (first parameter)
call Func2          ; Func2(var_14, var_C, var_8)
The last value pushed is the first argument. Read the pushes bottom-to-top to get the argument order.

Tracing Through a Caller

Once you have all function definitions, trace the caller by substituting:

; Given: Func4(a,b) = a/b, Func5(a,b) = a*b

var_14 = 192, var_10 = 1024, var_C = 256

var_8  = Func5(var_10, var_C)   = 1024 * 256     = 262144
var_8  = Func2(var_14, var_C, var_8) = 192 - (256 / 262144) = 192 - 0 = 192
var_4  = Func4(var_8, var_C)    = 192 / 256       = 0
var_C  = Func5(var_4, var_C)    = 0 * 256         = 0
...

Track each variable as it gets reassigned. Watch for integer division truncation (192 / 256 = 0, not 0.75).

Practice Tips

  • Always start with the simplest/leaf functions (ones that don't call anything else)
  • Name your arguments (a, b, c) and write the formula immediately
  • Use the Sandbox to verify individual function behavior with test values
  • Watch for: integer division truncation, signed vs unsigned operations, and functions that modify multiple variables

Control Structures

How high-level constructs like if/else, loops, and switch statements look in assembly.

If / Else

An if/else compiles to: compare, conditional jump over the "if" body, then an unconditional jump past the "else" body.

; C: if (eax > 10) { ebx = 1; } else { ebx = 2; }

    cmp eax, 10
    jbe else_branch     ; If eax <= 10, skip to else
    mov ebx, 1          ; "if" body
    jmp end_if
else_branch:
    mov ebx, 2          ; "else" body
end_if:

Pattern: CMP → conditional jump to else → if-body → JMP past else → else-body.

While Loop

A while loop is a conditional jump back to the top:

; C: while (ecx != 0) { eax += ecx; ecx--; }

loop_top:
    test ecx, ecx       ; Is ECX zero?
    jz loop_done         ; If zero, exit loop
    add eax, ecx         ; Loop body
    dec ecx
    jmp loop_top         ; Jump back to check
loop_done:

Pattern: label → test/compare → conditional jump out → body → JMP back to label.

Do-While Loop

A do-while puts the condition at the bottom (the body always runs at least once):

; C: do { eax += ecx; ecx--; } while (ecx != 0);

loop_top:
    add eax, ecx         ; Loop body (runs first)
    dec ecx
    test ecx, ecx
    jnz loop_top         ; If not zero, loop again

Pattern: label → body → test/compare → conditional jump back to label. This is the most common loop pattern in compiled code because it's more efficient (one jump instead of two).

For Loop

A for loop is just a while loop with an initializer and increment:

; C: for (ecx = 0; ecx < 10; ecx++) { eax += ecx; }

    mov ecx, 0           ; Initializer
loop_top:
    cmp ecx, 10          ; Condition
    jge loop_done         ; Exit if ecx >= 10
    add eax, ecx          ; Body
    inc ecx               ; Increment
    jmp loop_top
loop_done:

Compilers often optimize this into a do-while form with the check at the bottom.

Switch Statement (Jump Table)

Small switches compile to if/else chains. Large switches use a jump table — an array of addresses indexed by the switch value:

; C: switch(val) { case 0: ...; case 1: ...; case 2: ...; }

    mov eax, [val]
    sub eax, base_value   ; Normalize (subtract lowest case)
    cmp eax, max_case     ; Bounds check
    ja default_case       ; If out of range, go to default

    ; Jump table lookup:
    movzx edx, byte ptr [table + eax]  ; Load case index from table
    jmp [jump_targets + edx*4]         ; Jump to the case handler
In IDA, you'll see a byte array (the translation table) that maps input values to case indices, and a pointer array (the jump table) with the actual case addresses. The input value is used as an index into the byte array, and the result indexes into the jump table.

How to Identify Switch Tables in IDA

  1. Look for a SUB followed by a CMP + JA (bounds check + default case)
  2. The byte array after that maps input values → case numbers
  3. The case number indexes into a jump/address table
  4. Each entry in the jump table points to a case handler block

From your worksheet: the input value 0x77 has base subtracted (0x61), giving offset 0x16 = 22 decimal. Looking at index 22 in the byte array gives value 5, meaning it executes case 5 in the jump table.

Recognizing the Pattern

Endianness Guide

Endianness is about byte order in memory. It trips up everyone at least once.

The Core Idea

When you store a multi-byte value (like a 32-bit integer) in memory, the bytes have to go in some order. There are two conventions:

  • Little Endian: Least significant byte goes to the lowest address (x86 uses this)
  • Big Endian: Most significant byte goes to the lowest address (network byte order)

Memory trick: "Little end first" — the little end (low byte) of the number sits at the low address.

Why Does This Matter?

Endianness matters when you access part of a value or read raw memory:

SituationWhat happens
Store a DWORD, read a BYTE On x86 (little endian), reading the first byte gives you the low byte, not the high byte
Looking at hex dumps Memory dump shows EF BE AD DE but the actual DWORD value is 0xDEADBEEF — bytes are reversed!
Network protocols Network uses big endian. Receiving raw bytes on x86 requires byte-swapping (e.g., ntohs, ntohl)
Casting pointers *(char*)&myInt gives different bytes on LE vs BE systems

The Worksheet's Endianness Test

Your worksheet has this code that detects endianness. Let's walk through it step by step:

mov [ebp+var_4], 1              ; Step 1: Store 0x00000001 as a DWORD
movsx eax, byte ptr [ebp+var_4] ; Step 2: Read just the FIRST byte
cmp eax, 1                      ; Step 3: Is that first byte 0x01?
jnz short loc_401024            ; Step 4: If not, jump to Endian(B)
Step 1: Store the value 1 (= 0x00000001) at [ebp+var_4]. In memory:
var_40x01
+10x00
+20x00
+30x00
Because x86 is little endian, the low byte (0x01) is at the lowest address.
Step 2: movsx eax, byte ptr [ebp+var_4] reads only the first byte at that address. On little endian, the first byte is 0x01. So EAX = 1.
Step 3: cmp eax, 1 — is EAX equal to 1? Yes! So ZF = 1.
Step 4: jnz (jump if not zero) — ZF = 1, so the jump is NOT taken. We fall through to Endian(A).
Conclusion: Endian(A) = Little Endian. Endian(B) = Big Endian.
If it were big endian, the first byte would be 0x00 (the high byte), EAX would be 0, and JNZ would jump to Endian(B).
